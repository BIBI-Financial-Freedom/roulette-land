## 1. 아키텍처 개요

### 1.1 시스템 유형
- **SPA (Single Page Application)** — 화면 전환 JS 처리
- **클라이언트 중심** — UI·로직 브라우저 실행
- **서버리스 API 프록시** — LLM API 키 보호용 경량 프록시
- **정적 호스팅** — HTML/CSS/JS 파일 GitHub Pages 배포

## 2. 기술 스택

### 2.1 확정 기술

| 레이어 | 기술 | 버전 | 선택 이유 |
|--------|------|------|-----------|
| 마크업 | HTML5 | - | 표준 웹 |
| 스타일 | CSS3 | - | 네온 글로우, 애니메이션 |
| 스크립트 | Vanilla JavaScript | ES2022+ | 경량, 빌드리스 |
| 이미지 캡처 | html2canvas | 1.4+ | DOM→Canvas 렌더링 |
| 배포 | GitHub Pages | - | 무료 정적 호스팅 |
| API 프록시 | Cloudflare Workers / Vercel Edge | - | API Key 보호, 무료 티어 |

### 2.2 외부 API

| API | 용도 | 호출 빈도 |
|-----|------|-----------|
| OpenAI GPT-4o-mini / Gemini Flash | 팩폭 텍스트 생성 | 결과 생성 당 1회 |

### 2.3 미사용 기술 (의도적 배제)

| 기술 | 배제 이유 |
|------|-----------|
| React / Vue | 입력폼 + 결과 화면뿐, 프레임워크 불필요 |
| TypeScript | 빌드 없는 경량 프로젝트 |
| Backend (Node/Python) | 서버리스 프록시로 충분 |
| DB | 데이터 영속화 불필요 |

## 4. 모듈 설계

### 4.1 모듈 의존성

```
app.js (진입점)
  ├── ui/InputScreen.js
  │     └── utils/sanitize.js
  ├── ui/LoadingScreen.js
  ├── ui/ResultScreen.js
  │     ├── capture/CardCapture.js
  │     └── share/ShareManager.js
  ├── api/llm-client.js
  │     └── api/prompt-builder.js
  └── engine/ResultParser.js
        └── engine/FallbackEngine.js
```

### 4.2 모듈 책임

| 모듈 | 책임 |
|------|------|
| `app.js` | 화면 전환 관리, 전체 플로우 오케스트레이션 |
| `InputScreen.js` | 입력 폼 렌더링, 유효성 검사, 데이터 수집 |
| `LoadingScreen.js` | 로딩 애니메이션, 멘트 표시, 프로그레스 바 |
| `ResultScreen.js` | 결과 렌더링, 게이지 애니메이션, 버튼 핸들링 |
| `llm-client.js` | API 호출, 타임아웃, 에러 처리 |
| `prompt-builder.js` | 닉네임+소비 → 프롬프트 문자열 조합 |
| `ResultParser.js` | LLM 응답 텍스트 → 구조화된 객체 파싱 |
| `FallbackEngine.js` | API 실패 시 사전 정의된 팩폭 결과 랜덤 생성 |
| `CardCapture.js` | html2canvas로 결과 카드 → PNG 변환·다운로드 |
| `ShareManager.js` | 링크 복사, Web Share API 호출 |
| `sanitize.js` | XSS 방지 입력 새니타이징 |

## 5. LLM API 연동 설계

### 5.1 API 호출 흐름

```
[클라이언트]
     │
     │  POST /api/generate
     │  { nickname, expenses[] }
     │
     ▼
[서버리스 프록시]
     │
     │  1. Rate Limit 체크
     │  2. 입력 새니타이징
     │  3. 프롬프트 조합
     │  4. LLM API 호출 (API Key 포함)
     │  5. 응답 반환
     │
     ▼
[LLM API]
     │
     │  텍스트 생성 → JSON 응답
     │
     ▼
[클라이언트]
     │
     │  응답 파싱 → 결과 화면 렌더링
```

### 5.2 프록시 엔드포인트

```javascript
// POST /api/generate
// Request
{
  "nickname": "이현민",
  "expenses": [
    "탕후루 3만 원어치",
    "안 입는 요가복",
    "쓸데없는 구독 서비스"
  ]
}

// Response
{
  "success": true,
  "data": {
    "bankruptDate": "2026년 11월",
    "poorScore": 98,
    "futureDescription": "...",
    "roasts": ["...", "...", "..."],
    "summary": "..."
  }
}
```

### 5.3 Rate Limiting

| 항목 | 제한 |
|------|------|
| IP당 호출 | 10회/분 |
| 전체 호출 | 1,000회/일 |
| 초과 시 | 429 응답 + 폴백 결과 사용 |

## 7. 이미지 생성 파이프라인

### 7.1 캡처 프로세스

```
[결과 카드 DOM]
      │
      ▼
[html2canvas 렌더링]
      │  options: { width: 1080, height: 1920, scale: 2 }
      │
      ▼
[Canvas 객체 생성]
      │
      ▼
[canvas.toBlob('image/png')]
      │
      ▼
[Blob → Object URL]
      │
      ▼
[<a download> 트리거]
      │
      ▼
[PNG 다운로드 완료]
```

### 7.2 캡처 최적화

| 항목 | 설정 |
|------|------|
| 캡처 해상도 | 1080×1920 (scale: 2 → 실제 2160×3840 렌더링) |
| 폰트 로딩 대기 | `document.fonts.ready` 이후 캡처 |
| 그라데이션 보정 | CSS 그라데이션 호환 확인 |
| 네온 글로우 | box-shadow, text-shadow html2canvas 지원 확인 |
| 배경 | 투명 X → 검정 그라데이션 배경 고정 |

## 8. 이벤트 아키텍처

### 8.1 주요 이벤트 흐름

```
[입력 폼 Submit]
     │
     ├─ 유효성 검사 통과
     │     │
     │     ├─ 화면 전환: input → loading
     │     ├─ 로딩 애니메이션 시작
     │     └─ API 호출
     │           │
     │           ├─ 성공: ResultParser → 결과 렌더링
     │           │         화면 전환: loading → result
     │           │
     │           └─ 실패: FallbackEngine → 폴백 결과 렌더링
     │                     화면 전환: loading → result
     │
     └─ 유효성 검사 실패
           └─ 에러 메시지 표시

[이미지 저장 버튼]
     └─ CardCapture.capture() → PNG 다운로드

[공유 버튼]
     └─ ShareManager.share() → 링크 복사 / Web Share

[다시하기 버튼]
     └─ 상태 초기화 → 화면 전환: result → input
```

## 9. 보안 설계

### 9.1 클라이언트 보안

| 항목 | 대응 |
|------|------|
| XSS 방지 | 사용자 입력 새니타이징 (`sanitize.js`) |
| innerHTML 금지 | `textContent` 사용, 또는 DOM API로 생성 |
| API Key 노출 방지 | 키는 서버리스 프록시에만 저장 |
| CSP | Content Security Policy 헤더 설정 |

### 9.2 서버리스 프록시 보안

| 항목 | 대응 |
|------|------|
| CORS | 허용 도메인 화이트리스트 |
| Rate Limiting | IP 기반 요청 제한 |
| 입력 검증 | 닉네임 길이, 소비 텍스트 길이 서버측 재검증 |
| API Key | 환경 변수로 관리 |
| 프롬프트 인젝션 방지 | 사용자 입력을 프롬프트 데이터 영역에만 삽입 |

## 11. 성능 최적화

| 항목 | 방법 |
|------|------|
| 초기 로딩 | CSS/JS 최소화, 크리티컬 CSS 인라인 |
| 폰트 로딩 | `font-display: swap`, preload |
| html2canvas | 필요 시점에 동적 import |
| 이미지 | SVG 아이콘 사용, 래스터 이미지 최소화 |
| API 체감 속도 | 로딩 연출로 실제 대기 시간 마스킹 |
| 번들 크기 | html2canvas CDN 로드 (~40KB gzipped) |
