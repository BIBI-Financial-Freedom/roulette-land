## 1. 아키텍처 개요

### 1.1 시스템 유형
- **SPA (Single Page Application)** — 스텝별 화면 전환 JS 처리
- **클라이언트 중심** — UI·로직 브라우저 실행
- **서버리스 API 프록시** — LLM API 키 보호용 경량 프록시
- **정적 호스팅** — HTML/CSS/JS 파일 GitHub Pages 배포

## 2. 기술 스택

### 2.1 확정 기술

| 레이어 | 기술 | 버전 | 선택 이유 |
|--------|------|------|-----------|
| 마크업 | HTML5 | - | 표준 웹 |
| 스타일 | CSS3 | - | 네온 글로우, 애니메이션 |
| 스크립트 | Vanilla JavaScript | ES2022+ | 경량, 빌드리스 |
| 이미지 캡처 | html2canvas | 1.4+ | DOM→Canvas 렌더링 |
| 배포 | GitHub Pages | - | 무료 정적 호스팅 |
| API 프록시 | Cloudflare Workers / Vercel Edge | - | API Key 보호 |

### 2.2 외부 API

| API | 용도 | 호출 빈도 |
|-----|------|-----------|
| OpenAI GPT-4o-mini / Gemini Flash | 어휘 분석 + 팩폭 생성 | 분석 당 1회 |

### 2.3 미사용 기술 (의도적 배제)

| 기술 | 배제 이유 |
|------|-----------|
| React / Vue | 스텝 폼 + 결과 화면뿐, 프레임워크 불필요 |
| TypeScript | 빌드 없는 경량 프로젝트 |
| SSR (Next.js 등) | OG 태그는 URL 파라미터 + 클라이언트 JS로 처리 |
| Backend (Node/Python) | 서버리스 프록시로 충분 |
| DB | 데이터 영속화 불필요 |

## 5. LLM API 연동 설계

### 5.1 API 호출 흐름

```
[클라이언트]
     │
     │  POST /api/analyze
     │  { nickname, topic, text }
     │
     ▼
[서버리스 프록시]
     │
     │  1. Rate Limit 체크
     │  2. 입력 새니타이징 + 길이 검증
     │  3. 프롬프트 조합
     │  4. LLM API 호출 (API Key 포함)
     │  5. 응답 파싱 + 검증
     │  6. 응답 반환
     │
     ▼
[LLM API]
     │
     │  텍스트 분석 → JSON 응답
     │
     ▼
[클라이언트]
     │
     │  응답 파싱 → 결과 화면 렌더링
```

### 5.2 프록시 엔드포인트

```javascript
// POST /api/analyze
// Request
{
  "nickname": "이현민",
  "topic": "excuse",
  "text": "아 진짜 이거 제 잘못이 아니라 완전 상황이 그런 거예요..."
}

// Response
{
  "success": true,
  "data": {
    "vocabAge": 7,
    "score": 23,
    "repeatedWords": [
      { "word": "진짜", "count": 5 },
      { "word": "완전", "count": 3 },
      { "word": "대박", "count": 2 }
    ],
    "analysis": "...",
    "roast": "'진짜'를 빼면 말을 못 하는 7세 수준",
    "keyword": "복사붙여넣기급",
    "era": "2010년대"
  }
}
```

### 5.3 Rate Limiting

| 항목 | 제한 |
|------|------|
| IP당 호출 | 10회/분 |
| 전체 호출 | 1,000회/일 |
| 초과 시 | 429 응답 + 폴백 결과(TextAnalyzer 기반) |

## 7. OG 태그 시스템

### 7.1 OG 태그 전략

GitHub Pages는 정적 호스팅이므로, OG 태그를 서버에서 동적 렌더링할 수 없다.  
두 가지 방식을 병행한다:

**방식 A: 클라이언트 JavaScript (기본)**
- `index.html`에 기본 OG 태그 설정
- URL 파라미터 감지 시 `<meta>` 태그를 JS로 동적 수정
- 한계: 카카오톡/페이스북 크롤러가 JS를 실행하지 않으면 기본 OG만 표시

**방식 B: Cloudflare Workers HTML Rewrite (권장)**
- 프록시에서 HTML 응답을 가로채 OG 태그를 동적으로 수정
- 크롤러에게 완성된 OG 태그 전달 가능

### 7.2 OG Manager 동작

```javascript
// URL 파라미터 읽기
const params = new URLSearchParams(window.location.search);
const name = params.get('name');     // 닉네임
const age = params.get('age');       // 어휘 연령
const keyword = params.get('keyword'); // 키워드

// OG 태그 동적 수정
if (name && age) {
  document.querySelector('meta[property="og:description"]')
    .setAttribute('content', `${name} 님의 어휘력은 ${age}세 수준입니다`);
}
```

### 7.3 공유 URL 생성

```javascript
function generateShareURL(result) {
  const params = new URLSearchParams({
    name: result.nickname,
    age: result.vocabAge,
    keyword: result.keyword
  });
  return `${window.location.origin}${window.location.pathname}?${params}`;
}
```

## 9. 이벤트 아키텍처

### 9.1 주요 이벤트 흐름

```
[Step 1 — 닉네임 + 주제]
     │
     └─ [다음 버튼] → 유효성 검사 → Step 2 전환

[Step 2 — 텍스트 입력]
     │
     ├─ [입력 중] → 글자 수 카운터 업데이트
     │              → 30자 이상이면 제출 버튼 활성화
     │
     └─ [분석하기 버튼]
           │
           ├─ 유효성 검사 통과
           │     │
           │     ├─ 화면 전환: step2 → loading
           │     ├─ 로딩 애니메이션 시작
           │     └─ API 호출
           │           │
           │           ├─ 성공: ResultParser → 결과 렌더링
           │           │         화면 전환: loading → result
           │           │
           │           └─ 실패: FallbackEngine + TextAnalyzer
           │                     → 폴백 결과 렌더링
           │
           └─ 유효성 검사 실패
                 └─ 에러 메시지 표시

[결과 화면]
     ├─ [이미지 저장] → CardCapture.capture() → PNG 다운로드
     ├─ [공유하기] → ShareManager.share() → OG URL 생성 → 복사
     └─ [다시하기] → 상태 초기화 → step1 전환
```

## 10. 보안 설계

### 10.1 클라이언트 보안

| 항목 | 대응 |
|------|------|
| XSS 방지 | `sanitize.js`로 입력 새니타이징 |
| innerHTML 금지 | `textContent` 사용 + DOM API 생성 |
| API Key 노출 방지 | 키는 서버리스 프록시에만 저장 |
| URL 파라미터 인젝션 | OG 파라미터 값 이스케이프 + 길이 제한 |
| CSP | Content Security Policy 헤더 설정 |

### 10.2 서버리스 프록시 보안

| 항목 | 대응 |
|------|------|
| CORS | 허용 도메인 화이트리스트 |
| Rate Limiting | IP 기반 요청 제한 |
| 입력 검증 | 닉네임/텍스트 길이 서버측 재검증 |
| API Key | 환경 변수로 관리 |
| 프롬프트 인젝션 방지 | 사용자 입력은 프롬프트 데이터 영역에만 삽입 |

### 10.3 OG 파라미터 보안

| 항목 | 대응 |
|------|------|
| 파라미터 길이 | name: 최대 20자, keyword: 최대 30자 |
| 특수문자 | URL 인코딩 + HTML 이스케이프 |
| 스크립트 주입 | 파라미터를 `textContent`에만 삽입 |
