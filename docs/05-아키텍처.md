## 1. 아키텍처 개요

Operation Epic Fury는 **서버 없는 정적 SPA(Single Page Application)** 로, 모든 로직이 클라이언트 브라우저에서 실행된다.

### 설계 결정 근거

| 결정 | 이유 |
|------|------|
| 서버리스 | 트래픽 비용 0원, 서버 관리 불필요, 영구 무료 운영 |
| SPA | 화면 전환 시 깜빡임 없는 부드러운 경험 |
| 바닐라 JS 우선 | 번들 사이즈 최소화, 의존성 최소화, 빠른 로딩 |
| LocalStorage | 회원가입 없이 기록 저장, 개인정보 수집 없음 |

### 2.2 Game Logic Layer (게임 로직 계층)

핵심 게임 규칙과 상태를 관리한다. UI와 완전히 분리되어 독립적으로 테스트 가능하다.

```
src/
├── engine/
│   ├── SlotEngine.js        # 슬롯 엔진 (릴 결과 생성)
│   ├── ScoringEngine.js     # 점수 산출 + 조합 판정
│   ├── PlayerManager.js     # 참가자 CRUD + 순서 관리
│   └── RoundManager.js      # 라운드 진행 + 서든데스
├── models/
│   ├── Player.js            # Player 데이터 모델
│   ├── Symbol.js            # Symbol 데이터 모델
│   ├── SpinResult.js        # 스핀 결과 모델
│   └── RoundResult.js       # 라운드 결과 모델
└── constants/
    ├── symbols.js           # 심볼 정의 (이모지, 등급, 가중치)
    └── scoring.js           # 점수 테이블
```

#### SlotEngine

```
class SlotEngine:
  symbols: Symbol[]
  
  spin(): [Symbol, Symbol, Symbol]
    reel1 = weightedRandom(symbols)
    reel2 = weightedRandom(symbols)
    reel3 = weightedRandom(symbols)
    return [reel1, reel2, reel3]
```

#### ScoringEngine

```
class ScoringEngine:
  
  evaluate(symbols: [Symbol, Symbol, Symbol]): SpinResult
    matchType = detectMatchType(symbols)
    score = lookupScore(matchType)
    return { symbols, matchType, score, ... }
  
  detectMatchType(symbols): MatchType
    // 트리플 → 더블 → 싱글 순서로 체크
```

#### RoundManager

```
class RoundManager:
  players: Player[]
  currentPlayerIndex: number
  results: Map<Player, SpinResult>
  
  startRound()
  nextPlayer(): Player | null
  recordResult(player, spinResult)
  finishRound(): RoundResult
  checkSuddenDeath(): Player[] | null
  executeSuddenDeath(tiedPlayers): Player
```

#### RNG

```
class RNG:
  
  /**
   * 0 이상 max 미만의 암호학적 안전한 정수 반환
   */
  getSecureRandom(max: number): number
    array = new Uint32Array(1)
    crypto.getRandomValues(array)
    return array[0] % max
  
  /**
   * 가중치 기반 랜덤 선택
   */
  weightedSelect(items: {item, weight}[]): item
    totalWeight = sum(weights)
    random = getSecureRandom(totalWeight)
    // 누적합 방식으로 선택
```

#### AudioManager

```
class AudioManager:
  sounds: Map<string, AudioBuffer>
  isMuted: boolean
  
  preload(soundMap: {name: url}[])
  play(soundName: string, volume?: number)
  toggleMute()
```

## 3. 상태 관리

서버 없이 클라이언트에서만 동작하므로, 단순한 **중앙 상태 객체** 패턴을 사용한다.

### 3.1 GameState

```typescript
interface GameState {
  // 현재 화면
  currentScreen: 'start' | 'game' | 'suddenDeath' | 'result';
  
  // 참가자
  players: Player[];
  playerOrder: number[];  // 셔플된 인덱스 순서
  
  // 라운드
  currentPlayerIndex: number;
  results: Map<string, SpinResult>;  // playerId → SpinResult
  
  // 서든데스
  isSuddenDeath: boolean;
  suddenDeathRound: number;
  suddenDeathPlayers: Player[];
  
  // 최종 결과
  roundResult: RoundResult | null;
  
  // 설정
  isMuted: boolean;
}
```

## 7. 보안 고려사항

| 영역 | 대응 |
|------|------|
| **RNG 공정성** | `Math.random()` 대신 `crypto.getRandomValues()` 사용으로 예측 불가능한 난수 보장 |
| **입력 검증** | 참가자 이름 입력 시 HTML 이스케이프 처리 (XSS 방지) |
| **LocalStorage** | 민감 정보 저장하지 않음 (이름 + 점수만 저장) |
| **의존성** | 외부 라이브러리 최소화, CDN 사용 시 SRI(Subresource Integrity) 해시 적용 |
| **CSP** | Content-Security-Policy 헤더로 인라인 스크립트 제한 |

## 8. 성능 전략

| 전략 | 구현 |
|------|------|
| **번들 최소화** | 바닐라 JS로 프레임워크 종속 없음 (< 50KB) |
| **이미지 없는 심볼** | 시스템 이모지 사용으로 이미지 로딩 불필요 |
| **사운드 프리로드** | 게임 시작 시 모든 사운드 파일 비동기 로드 |
| **애니메이션** | CSS `transform` + `opacity`만 사용 (레이아웃 재계산 회피) |
| **requestAnimationFrame** | 릴 애니메이션에 rAF 사용으로 프레임 드랍 방지 |

**성능 목표:**
| 지표 | 목표 |
|------|------|
| FCP (First Contentful Paint) | < 1.5초 |
| TTI (Time to Interactive) | < 2.0초 |
| 릴 애니메이션 FPS | 60fps |
| 번들 사이즈 | < 500KB (사운드 포함) |

## 10. 테스트 전략

| 레벨 | 대상 | 도구 | 비고 |
|------|------|------|------|
| 단위 테스트 | SlotEngine, ScoringEngine, RNG, RoundManager | Vitest / Jest | 핵심 로직 100% 커버리지 목표 |
| 통합 테스트 | 라운드 전체 흐름, 서든데스 흐름 | Vitest / Jest | 시나리오 기반 |
| E2E 테스트 | 전체 사용자 흐름 | Playwright | 주요 해피 패스 |
| 수동 테스트 | 애니메이션, 사운드, 모바일 UX | — | 브라우저 실물 확인 |

**필수 테스트 케이스:**
- RNG 가중치 분포 검증 (10,000회 시뮬레이션)
- 모든 조합 유형의 점수 판정
- 서든데스 동점 반복 → 공동 패배 처리
- 참가자 2~10명 경계 테스트
